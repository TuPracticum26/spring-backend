# Document Version Management System Backend - Spring Boot API

Това е сървърната част на системата, прокектирана като RESTful API, отговорна за бизнес логиката, сигурността и комуникацията с базата данни.

## Технологичен стек
* **Framework:** Spring Boot 4.0.3 (Java 21)
* **Security:** Spring Security с **JWT (JSON Web Token)** - осигурява stateless автентикация
* **Persistence:** Spring Data JPA с MariaDB
* **Mapping:** MapStruct за разделяне на Entity слоя от DTO слоя
* **Documentation:** **SpringDoc OpenAPI** за автоматизиран Swagger интерфейс
* **Build Tool:** Maven

## Слоеве на архитектурата (Package Structure)
Проектът е организиран в логически слоеве за по-лесна поддръжка:
* **Controller:** Дефинира REST ендпоинтите и управлява HTTP заявките
* **Service:** Съдържа бизнес логиката и валидациите на статусите
* **Repository:** Интерфейси за комуникация с базата данни (JPA)
* **Entity:** JPA модели, отговарящи на таблиците в MariaDB
* **DTO:** Обекти за трансфер на данни, оптимизирани за фронтенда
* **Security:** Конфигурация на JWT филтри и права за достъп

## Сигурност и Роли
Достъпът до всеки метод е защитен на ниво контролер чрез анотациите на Spring Security:
* `ADMIN` Може да създва нови документи и нови версии. Пълен достъп до управление на потребители (User Management). Може да променя роли и да следи системни логове.
* `AUTHOR` Създава документи и нови версии. Има право да редактира съдържанието само на собствени чернови (DRAFT).
* `REVIEWER` Контрол на качеството. Единствената роля, която може да променя версия на статус от PENDING на APPROVED/REJECTED.
* `READER` Ограничен достъп. Може само да чете документи, които са финализирани и одобрени.

## Ключови API Ендпоинти
Проектът предоставя следните основни интерфейси за комуникация:

| Метод | Път | Описание | Достъп |
| :--- | :--- | :--- | :--- |
| `POST` | `/api/auth/login` | Генериране на JWT токен | Публичен |
| `GET` | `/api/documents` | Списък с всички документи | READER+ |
| `POST` | `/api/documents` | Създаване на нов документ | ADMIN, AUTHOR |
| `PATCH` | `/api/versions/{id}/status` | Смяна на статус (Review) | REVIEWER |
| `GET` | `/api/users` | Управление на потребители | ADMIN |

## Workflow (Жизнен цикъл)
Бакендът налага автоматизиран работен процес, който не позволява прескачане на етапи:

1. **DRAFT:** Версията се редактира от автора.
2. **PENDING:** Версията е изпратена за преглед.
3. **APPROVED:** Версията е одобрена, става видима за и читатели
4. **REJECTED:** Версията не е одобрена, но остава в историята на версиите на съответния документ.

## Управление на изключения (Exception Handling)
Бекендът използва глобален механизъм за обработка на грешки. При проблем, API-то връща стандартизиран отговор със съответния HTTP статус:

* **401 Unauthorized:** Невалиден или липсващ JWT токен.
* **403 Forbidden:** Потребителят няма права за съответното действие.
* **404 Not Found:** Търсеният ресурс (документ или версия) не съществува.
* **400 Bad Request:** Нарушение на бизнес правилата (напр. опит за одобрение на документ, който е все още DRAFT).

## Сигурност и CORS
* **JWT Filter:** Всяка заявка минава през филтър, който проверява валидността на токена в `Authorization` хедъра.
* **CORS:** Конфигуриран е достъп от `http://localhost:5173`, за да се позволи на React приложението да комуникира със сървъра без блокиране от браузъра.
